Threat Modeling

From the perspective of a Pentester.

  • Threat Modeling Manifesto
  • OWASP Threat Dragon
  • Drawio or Excalidraw - for quick sketch.

The Four Questions​

  • What are you working on? Understand the architeture.
  • What can go wrong? Identify threats.
  • What are you going to do about it? Mitigate, eliminate, transfer, or accept risk.
  • Did you do a good enough job? Review, validate, update.

Foundations​

English(Português).

  • Assets (Ativos): Anything of value.

    • Data assets: Information. Anything the system stores, processes and transmits. Personally Identifiable Information (PII), payment info, authentication credentials, transaction logs, to name a few.
    • System assets: Infraestructure that hosts and move the data. Servers, databases, APIs, network devices, and cloud services. The means to reach a data asset.
    • Business process assets: Workflows and processes that if disrupted make the company lose money.

  • Attack Surface (Superfície de Ataque): All potential target points to enter and extract data or interact with the system.

  • Data Flow Diagrams (DFD, Diagrama de Fluxo de Dados): Visual representation of how data moves.

    • External Entity (Entidade Externa): Rectangle -> User, system or service outside of control. Third-party data processor (e.g. payment), client browser.
    • Process (Processo): Circle -> App logic that acts on data. API gateways, webapps.
    • Data Store (Armazenamento de Dados): Parallel lines -> Where data persists at rest. SQL Database, Active Directory, log files.
    • Data Flow (Fluxo de Dados): Arrow -> Direction data moves between components. HTTP request, API call, database query.
    • Trust Boundary: Dashed line -> Border where trust changes. Internet - internal network.

Example:

[Client] -- HTTPS --> | (WebApp) -- API --> | (API gateway) -- SQL query --> || DATABASE ||

WebApp Login:

[user] -- POST --> | (login) -- check creds --> | (API) -- SQL query --> || DATABASE ||


[user] <-- OK /home --- | (login) <-- creds valid -- | (API) <-- query results || DATABASE ||

STRIDE​

Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege.

  • Helps to minimize inconsistency problems for threat identification.

Spoofing​

Violates Authentication

Threat actor pretends to be someone (or something) they are not. Authentication bypass with credential theft, session hijacking, IP (or MAC) spoofing, and forged authentication tokens.

Tampering​

Violates Integrity

Unauthorized modification of data. Intercepting and altering network traffic (AitM - T1557), manpulating database records, injecting malicious code, or modifying config files.

Repudiation​

Violates Non-Repudiation

System can't prove that a specific action was performed by a specific actor. Insufficient or easily trampered logs (makes incident response slower and forensics reconstruction difficult).

Information Disclosure​

Violates confidentiality

Sensitive data is exposed to unauthorized access. SQL injection data dumps, directory traversal can access configuration files (config.php), error messages and application stacks that reveal internal architecture, and insecure API responses with too much unecessary data.

Denial of Service​

Violates Availability

Make the system unresponsive or unavailable. DDoS, logic bombs (linux fork), targeted crashes with malformed input (sending an emoji to the database).

Elevation of Privilege​

Violates Authorization

Threat actors escalates from a lower privilege to a higher privilege, obtaining access to features and data they were never meant to reach. Broken access controls, kernel vulnerabilities, misconfigured permissions, and application logic flaws.

PASTA​

Process for Attack Simulation and Threat Analysis.

Define Objectives​

What does the business needs to protect and why.

Compliance, customer trust, uptime, data, ...

Define Technical Scope​

System, components and technologies.

Mapping the architecture, identifying attack surface, making an inventory of the components that supports the business objectives.

Guided by Scope and Rules of Engagement (RoE). Everything that appears here should be on Scope and outside of RoE.

Application Decomposition​

How does data move through the system, and where are the trust boundaries.

Identify processes, data stores, external entities, data flows, and trust boundaries. The entry points identified here become the initial target list.

Threat Analysis​

Threat intel based on client's industry, geolocation, and technology stack.

Gather information about who are the most probable threat actors (and what they target).

Vulnerability Analysis​

Gather vulnerabilities that exists in the target systems, and relate them to the identified threats.

Maps known vulnerabilities to the threats identified in Stage 4, Threat Analysis. CVE databases, prior vulnerability scan results, configuration reviews, and publicly disclosed issues affecting the client's technology stack. Here you prioritize what to test, instead of bruteforcing every vulnerability.

Attack Modeling​

How everything chain together into realistic attack paths.

Build attack trees to model a complete attack path, from initial access to final objective. Should be heavly based on the client's concerns.

"I want to know if an user can escape it's company context and get data from other companies that uses my SaaS".

Goal: Exfiltrate data from companies outside of current scope
|-- Path A: External
|    |-- Exploit SQL injections
|    |-- Attempt to bypass authentication
|-- Path B: With access
|    |-- Harvest app creds from public leaks
|    |-- Authenticate
|    |-- Horizontal privilege escalation
|    |-- Data dump
|-- Path C: Internal
|    |-- Harvest company employees creds
|    |-- Attempt to access internal network
|    |-- Attempt to pivot between networks
|    |-- Find database and query it

Risk and Impact Analysis​

Business impact if any of the attack paths succeeds.

Quantify business consequences if any of the attack paths succeeds. Its the executive summary of the report. Avoid technical terms, show the real impact. GPDR (LGPD) sanctions potential fines for non-compliance, customers notification, reputational, legal and operational impacts.

MITRE ATT&CK​

Adversarial Tactics, Techniques, and Common Knowledge.

Real world observations of adversary behavior.

https://attack.mitre.org/ Attack sim. based on Mitre - Atomic Red Team

  • Tactic: Adversary goal. Why the threat actor performs an action.
    • Credential Access (TA0006).
  • Technique: Method. The how.
    • Bruteforce (T1110).
  • Sub-technique: More specific variant of a technique.
    • Brute Force: Credential Stuffing (T1110.004).
