Kubernetes

Software Architectures:

  • Monolith -> Single service that do everything (users, payment, delivery).
    • When the demand is high, everything the whole thing needs to grow.
  • Microservices -> Divided system into its functions.
    • When the demand is high, only what is demanded need to grow.

Orchestrator of containers. In microservices, when an application starts receiving a lot of traffic, you start a new container with the app to distribute the traffic. When the peak is done, you go back to one container. This control is what Kubernetes does. Helps with availability and scalability.

  • Pod: Group of one or more containers. These containers can share storage and network resources. If the app needs to be scaled up, you increase the number of pods.
  • Cluster: Set of nodes.
  • Nodes: Workloads (applications) are run inside containers that are placed inside a pod. The pods run nodes. There's the control plane ( or master node) and worker nodes. Nodes can be virtual or physical machines. Nodes contain all the services necessary to run pods.

Control Plane​

Manages worker nodes and pods in the cluster.

  • Kube-apiserver: Front end of the control plane. Responsable for exposing the Kubernetes API.
  • Etcd: Key/value store with cluster data and its current state.
  • Kube-scheduler: Catch any new pods and assign it to a node.
  • Kube-controller-manager: Runs the controller processes (like node controller to watch nodes status). Interact with the scheduler node to schedule a new node to come up if a node is down.
  • Cloud-controller-manager: Communication between Kubernetes cluster and cloud provider API.

Worker Nodes​

  • Kubelet: Agent responsible for ensuring the containers are running in a pod. Executes actions given by the controller manager, like starting a pod with a container inside.
  • Kube-proxy: Network communication within the cluster. Traffic won't hit pods, they'll hit a Service associated with a group of pods, and then get directed to one of the associated pods.
  • Container runtime: Docker, rkt, runC. Enables pods to have containers.

Concepts​

  • Namespaces: It's a way to divide resources in isolated virtual groups. One group has its own resources and can't access other groups resources.
  • ReplicaSet: Set of replica pods. Guarantee the availability of a specific number of identical pods. They're managed by a deployment.
  • Deployment: Stateless management of RepplicaSets. When creating a deployment you can configure it to have a ReplicaSet with n number of pods.
  • StatefulSets: Like deployment, but used for applications that need a persistent identity and stable storage (like dbs, email). It guarantees that each pod gets a predictable name and its own dedicated storage volume.
  • Services: Placed in front of pods acting as an access point. This ensures pods have a single and the same IP and allows for requests to be load-balanced between pod replicas. ClusterIP, LoadBalancer, NodePort and ExternalName, to name a few.
  • Ingress: Manage external access to the services.

Configs​

  • apiVersion: Version of K8's API.
  • kind: Object (Deployment, Service, StatefulSet).
  • metadata: Data to identify the object.
  • spec: State of the object.

Service:

apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  selector:
    app: nginx
  ports:
    - protocol: TCP
      port: 8080
      targetPort: 80
  type: ClusterIP

Deployment:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:<version_tag>
        ports:
        - containerPort: 80

Commands​

# apply configs
kubectl apply -f deployment.yaml

# check status
kubectl get pods -n <namespace>

# show details of resource
kubectl describe pod <pod> -n <namespace>

# show logs | -c <container> if more then one in the pod
kubectl logs <pod> -n <namespace>

# shell into container
kubectl exec -it <pod> -n <namespace> -- sh

# port forward
kubectl port-forward service/<service_name> 8090:8080
