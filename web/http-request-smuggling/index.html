HTTP Request Smuggling

  • Mismatches in different web infrastructure components (front-end server treats the requests different from the back-end server) can make one request being mixed with other request when you manipulate the Content-Length and Transfer-Encoding headers.
  • Request splitting and HTTP desync are possible because of the nature of keep-alive connections and HTTP pipelining, which allow multiple requests to be sent over the same TCP connection. When calculating the sizes for Content-Length and Transfer-Encoding, it's crucial to consider the presence of carriage return \r and newline \n characters.

Smuggling primarly occurs due to discrepancies in how different servers (proxy and back-end) interpret HTTP request boundaries.

  • If both Content-Length and Transfer-Encoding are present, it's possible to crate ambiguities.
  • Some components prioritize Content-Length, others Transfer-Encoding.
  • Discrepancy can lead to one component believing the request has ended while another thinks it's still ongoing.

Be extreme carefull when testing on a production application. Smuggling can break it in many ways (cache poisoning, requests start failing, back-end pipeline fully desynced). I advise against it in prod.

  • PortSwigger - HTTP request smuggling
  • PayloadAllTheThings - Request Smuggling

Modern Apps​

  • Front-end server -> reverse proxy or load balancer.
  • Back-end server -> processes user requests, interacts with DBs, serves data to the front-end.
  • Databases -> persistent storage.
  • APIs -> interfaces that allow front and back-end servers to communicate.
  • Microservices -> small, independent services that communicate over a network.
  • Load Balancers -> distribute incoming network traffic across multiple servers to ensure that a single server doesn't receive too much traffic. Ensures high availability and reliability.
  • Reverse Proxies -> server app that accepts connection requests and forwards them to another server (or port on the same server), then sends the reply back to the original connection.
  • Caching -> store and reuse previously fetched data or computed results to speed up subsequent requests and computations.

HTTP Request Structure and Headers​

  • Only the ones that you'll need to understand to perform smuggling.

  • Content-Length: Request / response body size in bytes. Informs the receiving end how much data to expect.
  • Transfer-Encoding: Specify the form of encoding applied to the message body of an HTTP request / response. chunked indicates that the message is divided into a series of chunks, each preceded by its size in hex. Other values include compress, deflate, and gzip.
    • Mozilla Docs

Transfer-Encoding example, it will be useful to understand splitting:

POST /search HTTP/1.1
Host: normal-website.com
Content-Type: application/x-www-form-urlencoded
Transfer-Encoding: chunked

b
q=smuggling
0
  • b hex -> 11 bytes.
  • 0 -> end of the request for the Transfer-Encoding.

Manipulating this headers can confuse the proxy server.

HTTP 1​

CL.TE​

  • Content-Length/Transfer-Encoding -> The proxy uses Content-Lenght, the back-end uses Transfer-Encoding.
POST /search HTTP/1.1
Host: example.com
Content-Length: 130
Transfer-Encoding: chunked

0

POST /update HTTP/1.1
Host: example.com
Content-Length: 13
Content-Type: application/x-www-form-urlencoded

isadmin=true
  • The front-end server sees Content-Length of 130 and thinks it ends after isadmin=true. The back-end sees chunked and thinks it ends at 0. Basically you splitted the request in two, but only the back-end server will understand that.
  • This can lead to the back-end server treating the POST as a separate request.

TE.CL​

  • Transfer-Encoding/Content-Length -> The proxy uses Transfer-Encoding and the back-end uses Content-Length.
POST / HTTP/1.1
Host: example.com
Content-Length: 4
Transfer-Encoding: chunked

78
POST /update HTTP/1.1
Host: example.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 15

isadmin=true
0
  • Front-end server sees Transfer-Encoding: chunked and processes in chunks. 78 hex (120) indicates that the next 120 bytes are part of the request body and ends in 0. But the back-end server uses Content-Length, which is set to 4, processes the first 4 bytes (not including the entire smuggled POST), then interpret the POST as a separate request.

TE.TE​

  • Obfuscating the TE header.
  • Transfer-Encoding/Transfer-Encoding -> Both the front-end and beck-end server uses Transfer-Encoding. Crafting a malformed Transfer-Encoding header, the attacker aims to make one server ignore the header and use Content-Length instead. This can result in eather a CL.TE or TE.CL, depending on which server falls back to using the CL.
POST / HTTP/1.1
Host: example.com
Content-length: 4
Transfer-Encoding: chunked
Transfer-Encoding: chunked1

4e
POST /update HTTP/1.1
Host: example.com
Content-length: 15

isadmin=true
0
  • Two Transfer-Encoding headers, one is standard, the other is non-standard. Depending on server configuration, it might process the request based on the first header and ignore the malformed one. It relies on one of the servers handling the malformed, rejecting it and processing with the CL.

Transfer-Encoding : chunked

Transfer-Encoding: chunked
Transfer-Encoding: x

Transfer-Encoding:[tab]chunked

[space]Transfer-Encoding: chunked

HTTP 2​

  • RFC 9113
  • Requests and responses use a binary protocol instead of the human readable one from HTTP/1.
  • Each header is prefixed with its size for parsers to know how much info to expect.
  • Pseudo-headers are required for HTTP/2 requests. Then you have your regular Headers and Request body.
  • HTTP/1 uses \r\n to separate headers and : to separate the header name from the value. HTTP/2 doesn't need to do this, as it establishes precise boundaries for each part of the request or response.
  • Old HTTP headers, like Content-Length and Transfer-Encoding are meaningless with pure HTTP/2, but HTTP downgrades can occur. Load Balancers, proxies, CDNs can implement HTTP/2 but the server behind can still use HTTP/1.

Pseudo Headers​

You can check more about pseudo-headers in RFC 9113 - 8.3

Request​

  • :method -> HTTP method (GET, POST, ...)
  • :scheme -> HTTP and HTTPS are the most common, but can be non-HTTP schemes.
    • You can check more examples in RFC 3986 - 1.1.2.
  • :authority -> authority portion of the target URI. [ userinfo "@" ] host [ ":" port ]
    • More in RFC 3986 - 3.2.
  • :path -> path and query parts of the target URI. /dashboard.php?theme=dark

Response​

  • :status -> HTTP status code.

Desync​

  • HTTP/2 Downgrading -> A reverse proxy can serve content with HTTP/2, but requests it from the back-end using HTTP/1.1.
  • Sending an HTTP/2 request to the front-end server to influence the corresponding HTTP/1.1 request generated in the back-end causes an HTTP desync.

Expected:

H2.CL​

  • Proxy receives the HTTP/2 request on the front-end and passses the headers to the back-end. When the back-end reads the request, it interpret the Content-Length as valid and treats it as a POST request without a body. What comes after the headers will be interpreted as the start of a new request and since the request is not complete HTTP/1, the back-end will wait until more data arrives.
  • This leads to desync. If another user sends a request, it will be concatenated to our request.

  • Pay attention to the behavior of the random empty header X without \r\n.

H2.TE​

CRLF injection​

  • CRLF -> Carriage Return, Line Feed - \r\n (0xD and 0xA).
  • HTTP/2 handle binary info. Inserting any char in any field is possible.
  • Injecting \r\n might get translated by the proxy into HTTP/1 and will be interpreted as a header separator.

Request Tunneling​

Some proxy implementations have a single back-end connection to each user to separate requests.

  • Get info on how the back-end requests look.
  • HAProxy is vulnerable to CRLF (\r\n) injection.
POST /search HTTP/2
Host: <host>
User-Agent: Mozilla/5.0
Pwned: header

  • Send two requests in succession.
  • SHIFT + ENTER -> \r\n.
  • You can use this to try accessing a forbidden directory or file with a GET to /path.
    • GET /admin HTTP/1.1
  • You need to use Burp Suite inspector to do the injection.

Cache Poison​

  • You can make a cache poison attack if you find an upload feature, a part of the application that reflects content from a request parameter or an open redirect.
  • Attach a Pragma: no-cache header to the request to force the proxy to bypass cached content.

SSL cert to attach to the HTTPS server:

openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem -sha256 -days 3650 -nodes -subj "/C=XX/ST=StateName/L=CityName/O=CompanyName/OU=CompanySectionName/CN=CommonNameOrHostname"

HTTPS Server:

from http.server import HTTPServer, BaseHTTPRequestHandler
import ssl

PORT=8002

httpd = HTTPServer(('0.0.0.0', PORT), BaseHTTPRequestHandler)

context = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
context.load_cert_chain(certfile="cert.pem", keyfile="key.pem")

httpd.socket = context.wrap_socket(httpd.socket, server_side=True)

httpd.serve_forever()

h2c Smuggling​

  • For compatibility reasons, a web server can offer the client HTTP/1.1 and HTTP/2 connections.
  • HTTP/2 have two ways to negotiate connections:
    • h2: HTTP/2 over TLS.
    • h2c: HTTP/2 over cleartext.
      • The client sends a regular HTTP/1.1 request with Upgrade: h2c header. Compliant web servers will answer with 101 Switching Protocols. Proxy will tunnel the communications between client and server without checking the content, since it assumes the protocol changed to something other than HTTP.
      • Some proxies are aware of h2c and could try to handle the upgrade themselves.
  • h2c Smuggling only allows request tunnelling. It won't be possible to poison other users data.
